# 开发目标及实现方案（基本准则：最小改动/最小风险和成本）

## 1、文件展示：HTML 文件点击 → 新窗口渲染

### 改动范围

只改 3 个文件，沿用现有 `isImageFile` 模式：

| 文件 | 改动 | 行数 |
|------|------|------|
| `constants.ts` | 新增 `HTML_FILE_EXTENSIONS` | +2 行 |
| `fileTreeUtils.ts` | 新增 `isHtmlFile()` | +4 行 |
| `FileTree.tsx` | `handleItemClick` 里加一条 HTML 判断 | +8 行 |

### 实现逻辑

完全照搬 `isImageFile` → `selectedImage` → `ImageViewer` 的模式，但不弹 Modal，直接 `fetch → blob → window.open`：

```
用户点击 .html
  → handleItemClick
    → isHtmlFile(item.name) → true
    → fetch 文件内容
    → new Blob([html], { type: 'text/html' })
    → window.open(URL.createObjectURL(blob))
    → 页面加载后 URL.revokeObjectURL()
```

### 文件改动详情

**1. `constants.ts`** — 加扩展名集合
```
HTML_FILE_EXTENSIONS = new Set(['html', 'htm'])
```

**2. `fileTreeUtils.ts`** — 加判断函数
```
export function isHtmlFile(filename) {
  const ext = filename.split('.').pop()?.toLowerCase();
  return ext && HTML_FILE_EXTENSIONS.has(ext);
}
```

**3. `FileTree.tsx`** — 在 `handleItemClick` 里，图片判断之后、`onFileOpen` 之前插入
```
if (isHtmlFile(item.name)) {
  openHtmlInNewWindow(item.path, selectedProject.projectId);
  return;
}
```

`openHtmlInNewWindow` 写成一个独立函数（放在 FileTree.tsx 文件底部），负责 fetch + blob + window.open。

## 2、Shell 移动端输入优化

### 问题

移动端在 xterm.js 终端里直接打字体验很差：虚拟键盘遮挡、光标定位困难、中文输入法兼容性问题。

### 目标

对照参考项目 BLCLI_release_v1 的 InputBar 模式，增加原生 textarea 输入框 + `/` 命令按钮 + `@` 文件引用按钮 + Send/Stop 按钮。

> **注意**：禁用 xterm 终端键盘输入（captureInput 方案）已尝试但**已回退**。原因是终端创建是一次性的，`captureInput` 在终端创建之后变化无法重新绑定 `onData`，时序问题无法可靠解决。移动端终端仍可输入，但 ShellInputBar 提供了更友好的替代方式。

### 改动范围

| 文件 | 操作 | 说明 |
|------|------|------|
| `ShellInputBar.tsx` | **新增** | 移动端底部输入栏（~390 行） |
| `Shell.tsx` | 修改 | 渲染 ShellInputBar（`md:hidden`）；移动端不渲染快捷键面板 |

### 布局对比

```
当前移动端：                        目标移动端：
┌──────────────────────┐           ┌──────────────────────┐
│                      │           │                      │
│   xterm 终端          │           │   xterm 终端          │
│   （可输入+输出）      │           │   （可输入+输出）       │
│                      │           │                      │
├──────────────────────┤           ├──────────────────────┤
│ [黏贴][Esc][Tab]...   │           │ [/] [@] [textarea] [>]│
│ 快捷键面板（桌面端）   │           │ ShellInputBar（移动端） │
└──────────────────────┘           └──────────────────────┘
```

### ShellInputBar 设计

```
┌───────────────────────────────────────────┐
│ [/] [@] [___textarea___] [Send/Stop]     │
└───────────────────────────────────────────┘
```

| 元素 | 行为 |
|------|------|
| `/` 按钮 | 调用 `GET /api/providers/claude/skills?workspacePath=...` 加载技能列表，弹出浮层，点击技能将 trigger 插入 textarea。API 响应格式：`{ success: true, data: { skills: [...] } }` |
| `@` 按钮 | 调用 `GET /api/projects/:projectId/files` 获取完整文件树（maxDepth=10），弹出浮层。支持**目录导航**（点击文件夹进入子目录，`..` 返回上级）。选择文件后将 `@相对路径` 插入 textarea。API 返回嵌套 tree 数组（无 `{ success, data }` 外层包装） |
| textarea | 原生 textarea，自动扩展高度（max 160px），Enter 换行，点 Send 按钮发送 |
| Send/Stop 按钮 | 同一按钮两种状态：未发送时为蓝色 Send，发送后变为红色 Stop |

### `/` 按钮 — Skills 弹窗

- 数据来源：`GET /api/providers/claude/skills?workspacePath=...`
- 响应格式：`{ success: true, data: { skills: [{ name, description, trigger }] } }`
- 限制：仅首次打开时加载（`skillsLoaded` 标记），后续打开复用缓存

### `@` 按钮 — 文件选择器

- 数据来源：`GET /api/projects/:projectId/files` → 完整嵌套 tree（maxDepth=10）
- 响应格式：直接返回 `FileTreeNode[]` 数组（无外层包装），每个节点含 `{ name, path, type, children? }`
- path 为**绝对路径**（`path.join(dirPath, entry.name)`），插入时转换为**相对路径**（去除项目根路径前缀）
- **目录导航实现**：
  - `flattenAll()` 将嵌套 tree 递归打平为 `FileTreeNode[]`（含所有层级节点）
  - `getParentDir()` 提取父路径，通过 `currentDir` 状态过滤当前层级的直接子项
  - 路径全部归一化为 `/` 分隔符（Windows 下 `path.join` 产生 `\`，需统一）
  - 所有导航按钮使用 `onClick`（非 `onMouseDown`）以确保移动端 touch 事件兼容
- 限制：仅首次打开时加载（`filesLoaded` 标记），关闭浮层时重置 `currentDir` 到根目录

### Stop 按钮逻辑

1. 立即发送 `\x03`（Ctrl+C）停止当前 Claude 输出
2. 将上次发送的文本回填到 textarea，方便用户修改后重新发送
3. 1.5 秒后发送 `\x15`（Ctrl+U）清除 Claude Code 输入缓冲区（参考 BLCLI_release_v1）
4. 如果 1.5 秒内再次按 Stop，取消之前的定时器，重新计时

### 数据流

```
用户点击 Send
  → ShellInputBar.handleSend()
  → sendInput(text + '\r\n')
  → sendSocketMessage(ws, { type: 'input', data })
  → WebSocket → PTY shellProcess.write(data)
  → Claude CLI 处理 → 输出
  → xterm.write(output)
```

与桌面端的区别只有一步：桌面端是 `xterm.onData` → sendInput，移动端是 `ShellInputBar` → sendInput。后续链路完全一致。

### Shell.tsx 改动

- 新增 `isMobile` state：`matchMedia('(max-width: 767px)')` + change 事件监听
- 渲染 `ShellInputBar`（自带 `md:hidden` class，移动端可见）
- `TerminalShortcutsPanel` 条件渲染：`{!isMobile && <TerminalShortcutsPanel />}`

### 已回退的尝试

**captureInput 方案**（禁用终端键盘输入）：
- 在 `useShellRuntime` / `useShellTerminal` / `types.ts` 中新增 `captureInput` 参数
- 当 `captureInput === false` 时跳过 `xterm.onData` 绑定
- **失败原因**：终端在 `useShellTerminal` 的 effect 中一次性创建，`captureInput` 在创建之后的变化不会重新绑定 `onData`，且 effect 依赖过多会导致终端频繁重建
- **决策**：回退所有 captureInput 相关改动，移动端终端保持可输入

## 3、Shell 移动端触屏惯性滚动

### 问题

移动端在 xterm.js 终端中上下滑动滚动极慢。

### 根因分析

xterm.js 基于 Canvas 渲染，不经过浏览器原生 DOM 滚动。对比：

| 维度 | Chat 窗口 | Shell 终端 |
|------|----------|-----------|
| 渲染方式 | DOM 元素 | Canvas |
| 滚动机制 | 浏览器原生 overflow-y:auto | xterm 内部 viewport |
| 移动端加速 | 浏览器自动加速 touch→scroll | xterm 逐事件处理，无加速 |
| scrollSensitivity | 不适用 | 默认 1（每事件滚 1 行） |

浏览器对原生 DOM 滚动做了惯性/动量优化，但对 Canvas 内部的"滚动"一无所知。

### 方案选型

| 方案 | 描述 | 结论 |
|------|------|------|
| A. 提高 scrollSensitivity | 改 TERMINAL_OPTIONS 配置 | 治标不治本，touch→wheel 转换效率本身低 |
| B. 翻页按钮 | 快捷键面板加 Page Up/Down | 点按交互，"滑动"直觉不匹配 |
| **C. Touch 惯性滚动** | 接管 touch 事件 + RAF 动量动画 | **选中** — 参考项目 BLCLI_release_v1 验证过 |
| D. A+B+C 组合 | 三种叠加 | 过度设计 |

### 参考实现

移植自 `BLCLI_release_v1/web/src/components/OutputArea.tsx:135-247`，核心算法：

```
touchstart → 记录起点、取消旧动量
touchmove  → 计算帧间像素差 → term.scrollLines(lines) → 计算速度
touchend   → 启动 RAF 动量循环 (velocity *= FRICTION, 每帧)
```

关键常量（按 fontSize:14 调整 LINE_HEIGHT）：

| 常量 | 值 | 说明 |
|------|-----|------|
| LINE_HEIGHT_PX | 19 | fontSize:14 对应的行高（参考项目 fontSize:13 → 18） |
| FRICTION | 0.95 | 每帧速度衰减系数，~1 秒衰减至 5% |
| MIN_VELOCITY | 0.5 | 低于此值停止动画 |
| SWIPE_THRESHOLD | 8 | 8px 死区，区分点击和滑动 |

速度计算：`velocity = deltaY / dt_ms * 16`（归一化为每 16ms 帧的像素位移）

边界处理：
- 终端有文本选中时不接管 touch（`terminal.hasSelection()`），保留 xterm 原生选择能力
- 仅响应单指触摸（`e.touches.length === 1`），忽略多指手势
- `{ passive: false }` 注册事件以阻止浏览器下拉刷新等默认行为

### 改动范围

| 文件 | 操作 | 行数 |
|------|------|------|
| `constants.ts` | 新增 4 个 touch 滚动常量 | +5 行 |
| `useShellTerminal.ts` | 终端创建后注册 touch 事件 + 动量动画 | +~55 行 |

总计：**2 个文件，~60 行**。不改 UI 组件，不动布局，全部逻辑集中在 `useShellTerminal.ts` 一个 hook 内。

## 4、Shell 终端快速滚动条

### 问题

惯性滚动解决了滑动速度，但缺少一个可快速拖动定位的滚动条。面对 10000 行 scrollback buffer，从中间跳到顶部仍需多次滑动。

### 方案演进

| 阶段 | 方案 | 结论 |
|------|------|------|
| 1 | 自定义 TerminalScrollbar（绝对定位覆盖） | **废弃** — z-index 与 xterm canvas 冲突，touch 事件被终端截获 |
| 2 | CSS 样式化原生滚动条（`::-webkit-scrollbar`） | **废弃** — 移动浏览器用 overlay scrollbar，不支持 CSS 样式化 |
| **3** | **flex row 布局：终端 + 滚动条独立列** | **选中** — 滚动条独占一列，无 z-index 冲突，触摸可靠 |

### 布局

```
┌──────────────────────────┬──┐
│                          │█ │
│       终端 (flex-1)       │█ │ ← 滚动条 (16px, flex-shrink-0)
│                          │█ │   track + thumb
│                          │  │
└──────────────────────────┴──┘
```

关键区别：滚动条和终端是**兄弟元素**（flex row），各占各的列，不存在覆盖关系。

### 滚动条组件

| 属性 | 值 | 说明 |
|------|-----|------|
| 宽度 | 16px (w-4) | 手指触摸友好 |
| 轨道 | `bg-gray-800/60` | 始终可见 |
| 滑块 | `bg-gray-400/70` | 高度比例 = viewportRows / totalLines（最小 20px） |
| 位置同步 | `term.onScroll` + `term.onResize` | xterm 事件驱动 |
| 触摸 | `onTouchStart` 跳转 + `onTouchMove` 跟手 | `e.stopPropagation()` 防止穿透 |
| 显示 | `md:hidden` | 仅移动端 |

### 配套改动

- 隐藏 `.xterm-viewport` 原生滚动条（CSS `scrollbar-width: none` + `::-webkit-scrollbar { display: none }`）
- `html, body` 添加 `overflow: hidden` — 消除 shell 页面的全局滚动条（与参考项目一致）

### 改动范围

| 文件 | 操作 | 行数 |
|------|------|------|
| `TerminalScrollbar.tsx` | **新增** | ~80 行 |
| `Shell.tsx` | 修改 | +2 行 import，终端区改为 flex row |
| `index.css` | 修改 | +2 行（html/body overflow:hidden）+ 隐藏原生滚动条 |

总计：**3 个文件，~88 行**。
